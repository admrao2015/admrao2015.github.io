Структуры хранения данных • в проекте C-Store использовался движок Berkeley DB на основе LSM-дерева; • Microsoft SQL Server (для индексов columnstore), Db2 (для таблиц с опцией ORGANIZE BY COLUMN) и SAP IQ используют блочные структуры, где под дан- ные каждой колонки выделяются собственные экстенты (при этом экстент рассматривается как сплошная область хранения, не разделённая на стра- ницы): Экстент 786; 791; 799; 812 35.81; 33.10; 42.22; 13.02 Москва Москва; Казань; Казань Илья; Антон; Мария; Анна • в MonetDB используются массивы в памяти, отображаемые на файлы при помощи механизма memory-mapped ﬁles; • экспериментальный движок Zedstore, обеспечивающий колоночное хране- ние для PostgreSQL, хранит колонки в виде B-деревьев: каждой записи при вставке присваивается последовательно возрастающий внутренний иден- тификатор, по которому упорядочены значения атрибутов в каждом дереве. Некоторые СУБД используют колоночные структуры не как основное хранили- ще, а как дополнительное представление данных, хранящихся построчно. Так, на- пример, дополнение Oracle In-memory Option, доступное начиная с версии 12cR1, создаёт отдельную область в памяти, вне буферного кеша (in-memory columnar storage). В этой области данные, загружаемые из обычных таблиц, сохраняются в колоночном формате. Система не отказывается и от буферного кеша, где стра- ницы по-прежнему хранятся в том же формате, что и на диске: 786; Илья; 35.81; Москва 791; Антон; 33.10; Казань 799; Мария; 42.22; Казань 812; Анна; 13.02; Москва 786; Илья; 35.81; Москва 799; Мария; 42.22; Казань 35.81; 33.10; 42.22; 13.02 Москва; Казань; Казань; Москва буферный кеш дополнительное колоночное хранилище 106
